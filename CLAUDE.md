## Project Overview

FLARE is a custom STM32H7-based quadcopter flight controller with ESP-NOW RC link. Three firmware targets: STM32H723 FC (CMake), ESP32 quad receiver (PlatformIO), ESP32 remote transmitter (PlatformIO).

## Build & Flash

### STM32H723 Flight Controller

```powershell
# From firmware/fc/
cmake --build --preset Debug
# Output: build/Debug/fc.elf

# Flash (ST-Link V2, SWD — CLI only, GUI connection unreliable)

```

## Architecture

**Control loop (100Hz, STM32H723):** IMU read → sensor fusion → RC parse → PID → DSHOT motor output → UART telemetry.

**RC link:** Remote ESP32 (50Hz) → ESP-NOW → Quad ESP32 → UART (115200) → STM32 USART2 (PA3).

**Shared protocol:** `firmware/shared/flare_protocol.h` — 14-byte RC packet (magic 0xAF + 4× uint16_t channels + armed/mode + CRC-8/MAXIM).

### Key Source Files

| File | Role |
|------|------|
| `firmware/fc/Core/Src/main.c` | Boot sequence + 100Hz control loop |
| `firmware/fc/Core/Src/dshot.c` | DSHOT300 DMA burst driver |
| `firmware/fc/Core/Src/imu_fusion.c` | Complementary filter (roll/pitch/yaw) |
| `firmware/fc/Core/Src/flare.c` | Motor mixing + arming state machine |
| `firmware/fc/Core/Src/rc.c` | USART2 interrupt-driven RC receiver |
| `firmware/fc/Core/Src/mag.c` | QMC5883P I2C driver |
| `firmware/shared/flare_protocol.h` | RC packet definition |

### CubeMX-Generated Files

`firmware/fc/Core/Src/{spi,i2c,usart,gpio,tim,dma}.c` are regenerated by STM32CubeMX from `fc.ioc`. Custom code **must** live inside `/* USER CODE BEGIN */` / `/* USER CODE END */` markers to survive regeneration.

## Firmware Invariants

- **Throttle mapping (`main.c`):** RC channel (1000–2000) maps to DSHOT (48–2047): `dshot = ((rc - 1000) / 1000.0) * (2047 - 48) + 48`. The floor is 48, not 0 — DSHOT values 1–47 are reserved BLHeli_S special commands.
- **PID spool-up threshold (`flare.c`):** `FLARE_Update()` sends equal throttle to all motors (no PID) when `state.throttle < 250`. This is ~10% above the DSHOT minimum, giving ESCs time to spool before stabilisation engages.
- **Arm switch logic (`esp32_remote`):** `PIN_ARM_SWITCH` (D12) is `INPUT_PULLUP` — `LOW = armed`, `HIGH = disarmed`. Always arm with throttle stick at physical minimum (display ~0%) to avoid spin-up jolt.
- **Magnetometer interference:** Running motors corrupt the QMC5883P heading via EMI. Yaw fusion drifts when motors spin without calibrated hard-iron offsets. Do not rely on yaw during bench motor tests.

## Hardware Quirks

- **STM32H7 SPI:** 3-byte transactions truncated by hardware — always use 4-byte reads from BMI323.
- **BMI323 Feature Engine:** Must be initialized before ACC_CONF/GYR_CONF mode bits are accepted (undocumented).
- **QMC5883P:** This is the P variant (I2C addr 0x2C, different register map) — not the L variant commonly documented. No FBR register at 0x0B.
- **DSHOT DMA:** `HAL_TIM_DMABurst_WriteStart` ignores NDTR on STM32H7 — use direct register writes. Buffer in D1 AXI SRAM (0x24000000).
- **ST-Link V2 clone:** Use CLI only with `freq=100 reset=HWrst`; GUI connection unreliable; ITM SWO non-functional.
- **BLHeli_S ESC keepalive:** Disarms after ~250ms without a valid DSHOT frame — every control loop iteration must call `DSHOT_SendThrottle`.
- **DSHOT DMA vs USART2 priority:** DMA1_Stream0 at priority 0,0; USART2 at 8,0 — prevents RC byte loss during DSHOT bursts.
